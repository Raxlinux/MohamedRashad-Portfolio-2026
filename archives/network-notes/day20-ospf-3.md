# Day 20: OSPF Network Types and LSA Fundamentals

OSPF is designed to be flexible across different physical media. To optimize how updates are sent, OSPF uses different **Network Types**. Each type changes how routers discover neighbors and whether they need to elect a "leader" (DR) to manage the segment.

---

## 1. OSPF Network Types Overview

There are three primary network types you will encounter in OSPF configurations:

1. **Broadcast:** The default for Ethernet and FDDI.
2. **Point-to-Point (P2P):** The default for Serial interfaces using PPP or HDLC.
3. **Non-Broadcast (NBMA):** Used for multi-access networks that don't support broadcasting (like Frame Relay).

---

## 2. The Broadcast Network Type

This is the most common type used in modern LANs. Because many routers can exist on a single Ethernet segment, OSPF uses a hierarchy to prevent the network from being flooded with redundant updates.

### Neighbor Discovery & Roles

* **Multicast Discovery:** Routers dynamically find each other using the multicast address **224.0.0.5**.
* **The DR and BDR:** To keep the network efficient, routers elect a **Designated Router (DR)** and a **Backup Designated Router (BDR)**.
* **DROther:** Any router that is not the DR or the BDR is known as a **DROther**.

### The Election Process

The election occurs during the **2-Way** state based on a specific order of priority:

1. **Highest OSPF Interface Priority:** (Default is 1. A priority of 0 means the router can never become a DR).
2. **Highest OSPF Router ID (RID):** Used as a tie-breaker if priorities are equal.

> [!Important]
> **Non-Preemptive Nature:** OSPF elections are non-preemptive. If a router with a higher priority joins the network *after* a DR has already been elected, it will **not** take over the DR role. If the DR fails, the BDR is promoted to DR, and a new election is held for the BDR spot.

### Adjacency and LSA Flooding

* **Full Adjacency:** DROthers only form a "Full" adjacency with the **DR** and **BDR**.
* **DROther Communication:** DROthers stay in the **2-Way** state with each other; they do not exchange LSAs directly.
* **Traffic Flow:** Updates to the DR/BDR are sent via multicast **224.0.0.6**. The DR then distributes these updates to everyone else via **224.0.0.5**. This ensures everyone has a synchronized **LSDB** (Link State Database) while minimizing traffic.

---

## 3. Point-to-Point (P2P) Network Type

This type is used when only two routers are connected directly, such as over a Serial link.

* **Encapsulation:** Enabled by default on Serial interfaces using **HDLC** or **PPP**.
* **No DR/BDR:** Since there are only two routers, there is no need for a "boss." No election is held.
* **Adjacency:** Both routers form a direct **Full** adjacency and exchange LSAs immediately.
* **Discovery:** Neighbors are still discovered dynamically using **224.0.0.5**.

---

## 4. OSPF Adjacency Requirements

For two routers to become neighbors and share LSAs, the following parameters **must** match on the connecting interfaces:

* **Area ID:** Routers must be in the same OSPF area.
* **Subnet:** Interfaces must be on the same primary IP subnet and mask.
* **Hello & Dead Timers:** If one router expects a Hello every 10 seconds and the other sends it every 30, they will not form an adjacency.
* **Authentication:** Passwords and types (Plaintext or MD5) must match.
* **MTU:** The Maximum Transmission Unit must be identical to prevent database exchange issues.
* **Network Type:** You cannot form a neighbor relationship between a Broadcast interface and a Point-to-Point interface.
* **Unique Router ID:** Both routers must have different RIDs.
* **Status:** The OSPF process must be active (not shut down).

---

## 5. OSPF LSA Types

The **Link-State Database (LSDB)** is composed of various **Link-State Advertisements (LSAs)**. While there are 11 types, these four are the most critical:

| Type | Name | Description |
| --- | --- | --- |
| **Type 1** | **Router LSA** | Generated by every router to describe its own links and state. |
| **Type 2** | **Network LSA** | Generated by the **DR** to describe all routers on a broadcast segment. |
| **Type 3** | **Summary LSA** | Generated by an **ABR** to describe routes to other areas. |
| **Type 5** | **External LSA** | Generated by an **ASBR** to describe routes redistributed from outside OSPF. |

## 6. Configuration & Verification Commands

### Interface Settings

```bash
# Change the OSPF priority for DR/BDR election (0-255)
Router(config-if)# ip ospf priority 200

# Manually change the OSPF network type
Router(config-if)# ip ospf network {broadcast | point-to-point | non-broadcast}

# Adjust OSPF Timers
Router(config-if)# ip ospf hello-interval 10
Router(config-if)# ip ospf dead-interval 40

# MTU and Authentication
Router(config-if)# ip mtu 1500
Router(config-if)# ip ospf authentication message-digest
Router(config-if)# ip ospf message-digest-key 1 md5 PASSWORD

```

### Serial Interface Commands

```bash
# Change serial encapsulation to PPP (Default is HDLC)
Router(config-if)# encapsulation ppp

# Set the speed for back-to-back lab connections (DCE side only)
Router(config-if)# clock rate 128000

```

### Verification

```bash
# Check OSPF interface status, network type, and DR/BDR roles
Router# show ip ospf interface brief

# View the puzzle pieces (LSAs) that make up the LSDB
Router# show ip ospf database

# View hardware information and see if the cable is DCE or DTE
Router# show controllers s0/0/0

```

---

> [!Tip]
> If you see an OSPF neighbor stuck in the **2-Way** state on an Ethernet link, don't panic! This is normal behavior for two **DROther** routers. They only reach the **Full** state with the DR and BDR.
